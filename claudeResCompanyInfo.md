# [회사정보] 저장 로직 작업 요청서

## 1. 작업 개요
이제 회사정보(`env`)의 **저장 로직 작업**을 진행합니다.

이번 작업에서는 아래 원칙을 기준으로 구현해 주세요.

- **입력폼 저장**과 **이미지 저장/삭제**는 서로 별개로 처리할 것
- 작업 전, **궁금한 점이 있으면 먼저 질문**할 것
- 더 나은 구현 방향이 있다면 **근거와 함께 제안**할 것

---

## 2. 공통 작업 원칙

### 2-1. 저장 전 확인
- 교정관리와 마찬가지로, 저장/삭제와 같은 주요 작업은 **`gMessage` 기반 confirm 확인 후 진행**할 것
- 현재 어드민 페이지에서 해당 공통 confirm 로직이 없다면, **어드민 페이지용 공용 함수(adminCommon.js 등)를 생성하여 처리**할 것

### 2-2. 예외 처리
- 서버 오류 또는 기타 오류 발생 시, **클라이언트에 오류 메시지를 반드시 보여줄 것**
- 프론트/백엔드 모두 **예외처리를 명확히 구현**할 것
- 단순 콘솔 출력으로 끝내지 말고, **사용자가 인지할 수 있는 형태로 메시지 처리**할 것

### 2-3. 저장 후 처리
- 저장 성공 시, **회사정보 페이지를 재렌더링하여 최신 정보가 반영되도록 할 것**
- 이미지의 경우도 **DB에 저장된 경로 기준으로 다시 조회하여 화면에 표시**할 것

### 2-4. 로그 처리
- 아래 모든 동작에 대해 **로그를 항상 남길 것**
  - 입력폼 저장
  - 이미지 저장
  - 이미지 삭제

---

## 3. 입력폼 저장 요구사항

### 3-1. 기본 처리
- 필수값은 없음
- 단, 모든 입력값은 **좌우 공백(trim)을 제거한 상태로 서버에 전달**할 것

### 3-2. 전화번호 / 휴대폰연락처 처리
대상 필드:
- 전화번호
- 휴대폰연락처

요구사항:
- 값이 존재할 때만 검증 수행
- **정규식을 통해 형식 검증**할 것
- 사용자가 입력할 때 **숫자만 입력 가능**하도록 처리할 것
- 입력 중 자동으로 **`-` 하이픈이 생성**되도록 할 것

참고:
- 이와 관련된 함수가 **교정관리의 `common.js` 에 존재**
- 필요 시 **`adminCommon.js` 에 선언하여 재사용**하거나,
- 어드민 React 구조에 맞게 별도로 구현해도 됨
- 단, 최종적으로는 **일관된 UX로 동작**해야 함

### 3-3. 이메일 검증
- 이메일은 값이 존재할 때만 검증 수행
- 저장 시 **이메일 형식에 대한 정규식 검증**을 할 것

### 3-4. 사업자등록번호 검증
- 사업자등록번호는 값이 존재할 때만 검증 수행
- 저장 시 **사업자번호 형식에 대한 정규식 검증**을 할 것

## 4. 이미지 저장/삭제 요구사항

대상 이미지:
- `kolas`
- `ilac`
- `company`

### 4-1. 업로드 조건
- 각 이미지 항목은 **단일 파일만 업로드 가능**
- **이미지 확장자만 허용**
- 이미지당 **최대 10MB까지만 허용**

### 4-2. 저장 방식
- 업로드한 원본 파일명과 무관하게, 오브젝트 스토리지 저장 시 아래 규칙을 따를 것:
  - `kolas.확장자`
  - `ilac.확장자`
  - `company.확장자`

예시:
- 사용자가 `my_kolas_logo.png` 를 업로드해도 실제 저장 파일명은 `kolas.png`

### 4-3. DB 반영
- 오브젝트 스토리지 업로드 후,
- `env` 테이블에는 **해당 저장 경로(path)** 를 저장할 것

### 4-4. 기존 이미지 교체
- 이미지가 이미 존재하는 상태에서 다른 이미지를 선택해 저장할 경우,
- 오브젝트 스토리지의 기존 파일은 **새 이미지로 교체**될 것
- DB 값은 **동일한 경로를 유지하는 구조여도 괜찮음**

즉:
- 파일 실체만 교체
- DB에는 동일한 logical path가 유지될 수 있음

### 4-5. 삭제 처리
- 이미지의 `삭제` 버튼 클릭 시, **`gMessage` confirm 확인 후 진행**할 것
- 삭제 시 아래 두 가지를 모두 수행해야 함
  1. DB에서 해당 이미지 경로 값을 **빈값으로 업데이트**
  2. 오브젝트 스토리지에서도 해당 이미지 파일을 **실제로 삭제**

### 4-6. 화면 표시
- 저장 성공 후에는 최신 데이터를 다시 조회하여,
- **DB에 저장된 이미지 경로 기준으로 이미지가 표시**되어야 함
- 이미지가 표시될 때는 **바깥 박스 크기를 넘지 않도록** 처리할 것

## 5. 구현 시 고려사항

### 5-1. 프론트 검증 + 서버 검증
가능하면 아래 방향으로 구현해 주세요.
- 프론트에서 1차 검증
- 서버에서 2차 검증

즉, 클라이언트 UX를 위해 프론트 검증을 하되,
최종 저장 안전성을 위해 서버에서도 동일한 검증을 보장해 주세요.

### 5-2. 공통 함수 정리
어드민 페이지에서도 반복되는 아래 기능은 공통화 여부를 검토해 주세요.
- confirm 메시지 처리
- 에러 메시지 처리
- 전화번호 자동 하이픈 처리
- 입력값 trim 처리
- 파일 검증 처리

공통화가 가능하면 적절한 위치에 정리해 주세요.

### 5-3. 저장 성공 후 상태 동기화
입력폼 저장 / 이미지 저장 / 이미지 삭제 후에는
단순 화면 값만 바꾸는 방식보다,
가능하면 **재조회 후 화면을 최신 상태로 맞추는 방식**을 우선 고려해 주세요.

## 6. 구현 요청 사항 정리

### 입력폼 저장
- 입력폼 저장은 이미지 저장과 별개
- 필수값 없음
- 모든 입력값 trim 후 저장
- confirm 후 저장
- 전화번호 / 휴대폰연락처 형식 검증
- 전화번호 / 휴대폰연락처 숫자만 입력 + 자동 하이픈 처리
- 이메일 형식 검증
- 사업자등록번호 형식 검증
- 오류 발생 시 사용자 메시지 표시
- 성공 시 재조회 및 재렌더링
- 로그 기록

### 이미지 저장
- 이미지별 개별 저장
- 단일 파일만 허용
- 이미지 파일만 허용
- 최대 10MB 제한
- 오브젝트 스토리지 파일명은 `kolas`, `ilac`, `company` 고정 규칙 사용
- `env` 에는 저장 경로 반영
- 기존 이미지 존재 시 새 파일로 교체
- 성공 시 재조회 및 재렌더링
- 박스 영역 넘지 않도록 표시
- 로그 기록

### 이미지 삭제
- confirm 후 삭제
- DB 값 빈값 처리
- 오브젝트 스토리지 파일 삭제
- 성공 시 재조회 및 재렌더링
- 로그 기록

## 7. 작업 전 클로드가 먼저 확인하거나 제안해줄 사항
작업 시작 전, 아래 중 확인이 필요한 것이 있다면 먼저 질문해 주세요.

- 빈값 저장 시 `null` 로 할지 `""` 로 할지
    => DB 해당 컬럼의 타입에 따라 알아서 할 것
- 이미지 URL 표시 방식 (절대경로 / 상대경로 / presigned URL 등)
  => 중요하진 않고, 해당 경로의 이미지만 표시되면 됨.  직원수정(memberModify)에서 성적서서명이미지를 보여주는 방식 참고할 것

또한, 더 나은 방향이 있다면 아래 형식으로 제안해 주세요.

- 현재 요구사항 기준 구현안
- 더 나은 대안
- 왜 더 나은지
- 적용 시 영향 범위
